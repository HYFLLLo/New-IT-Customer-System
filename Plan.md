## 1. 项目概述

### 1.1 项目背景

随着企业IT支持请求量的不断增长，传统人工响应模式面临响应慢、知识传承依赖个人经验、知识库更新滞后等痛点。本项目旨在构建一套**AI驱动的IT智能问答与工单管理系统**，通过 RAG（检索增强生成）技术实现基于企业知识库的智能问答，同时保留人工介入通道处理复杂问题，实现"AI先行、人工兜底"的IT服务模式。

### 1.2 项目定位

本系统定位为**企业内部IT服务台**，面向全体员工提供：
- **7×24小时即时响应**的AI问答服务
- **可视化工单管理**的全生命周期跟踪
- **可运营知识库**的持续积累与优化
- **服务质量分析**的量化评估体系

### 1.3 核心价值

| 价值维度 | 具体体现 |
|---------|---------|
| **效率提升** | AI即时响应，减少80%以上简单重复问题的等待时间 |
| **知识沉淀** | 将IT工程师经验转化为可复用的知识资产 |
| **服务透明** | 工单状态实时可见，告别"报修后杳无音讯" |
| **持续优化** | 通过反馈闭环不断优化AI回答质量 |

### 1.4 目标用户

| 用户角色 | 使用场景 | 核心诉求 |
|---------|---------|---------|
| **普通员工** | 提问IT问题、查看工单状态、评价服务 | 快速得到答案、过程透明 |
| **IT坐席/Agent** | 处理工单、回复用户、知识库维护 | 高效协作、质量可控 |
| **IT管理员** | 知识库管理、报告生成、数据分析 | 整体把控、持续优化 |

## 2. 核心目标

### 2.1 业务目标

1. **AI问答准确率**：基于知识库的问答，confidence score ≥ 0.8 的回答直接呈现
2. **工单解决率**：目标首响时间 < 30分钟，整体解决率 > 90%
3. **知识库覆盖率**：覆盖企业IT常见问题 > 95%
4. **用户满意度**：满意度评分 ≥ 4.2/5.0

### 2.2 技术目标

1. **RAG问答链路**：用户提问 → 向量检索 → 上下文拼接 → LLM生成 → 置信度评估 → 响应/建单
2. **混合检索能力**：融合向量相似度（ChromaDB）与关键词匹配（BM25）的混合检索
3. **多格式文档解析**：支持 PDF、Markdown、DOCX 格式的知识库文档自动解析入库
4. **可扩展架构**：支持未来接入更多LLM Provider、知识源和数据 Sink

### 2.3 非功能目标

| 指标 | 目标值 |
|-----|-------|
| 系统可用性 | 99.5% |
| API响应时间 | P95 < 3s |
| 知识库检索延迟 | < 500ms |
| 支持并发用户 | ≥ 100 并发 |

## 3. 技术栈选型

### 3.1 技术选型原则

- **成熟优先**：选用经过大规模生产验证的技术栈，降低技术风险
- **成本效益**：优先采用开源或低成本方案，核心能力按需付费
- **团队适配**：技术栈在团队技术能力范围内，便于维护
- **生态完善**：周边工具链完备，利于长期发展

### 3.2 前端技术栈

| 类别 | 选型 | 版本 | 选型理由 |
|-----|------|------|---------|
| 框架 | Next.js (App Router) | 16.2.1 | React生态最成熟的全栈框架，API Routes替代传统BFF |
| 语言 | TypeScript | 5.x | 类型安全，IDE支持好，降低运行时错误 |
| UI库 | Radix UI + Tailwind CSS | - | 原子化CSS + 无样式Headless组件，定制灵活 |
| 状态管理 | React Server Components + URL State | - | Next.js App Router模式下减少客户端状态 |
| 通知 | Sonner | 2.0.7 | 轻量、React友好的Toast通知 |
| 图表 | (预留，可扩展) | - | 后期如需dashboard可视化 |

**备选方案：**
- UI框架曾考虑shadcn/ui，但其本质也是Radix UI + Tailwind，当前方案等效
- 状态管理如遇复杂场景可引入Zustand

### 3.3 后端技术栈

| 类别 | 选型 | 版本 | 选型理由 |
|-----|------|------|---------|
| 运行时 | Node.js (Next.js API Routes) | LTS | 统一前端后端语言，减少上下文切换 |
| 认证 | JWT (jose库) | 6.2.2 | 轻量、无状态，适合API认证 |
| 密码 | bcryptjs | 3.0.3 | 业界标准密码哈希 |

**备选方案：**
- 如需更复杂权限管理可引入Casbin或OPAL
- 如需WebSocket实时通信可引入Socket.io/Pusher

### 3.4 数据库与存储

| 类别 | 选型 | 版本 | 选型理由 |
|-----|------|------|---------|
| 关系数据库 | PostgreSQL | 17 | 可靠性高，JSONB支持灵活扩展字段 |
| ORM | Prisma | 6.19.2 | 类型安全、迁移方便、IDE支持好 |
| 向量数据库 | ChromaDB | 3.4.0 | 轻量、部署简单、支持混合检索（BM25+向量） |
| 文件存储 | 本地文件系统 | - | 初期场景简单，无需S3等对象存储 |

**备选方案：**
- 如数据量增长可迁移至Supabase（自带PG + Auth + Storage）
- 向量库如需更强能力可考虑Qdrant/Pinecone

### 3.5 AI/LLM 集成

| 类别 | 选型 | 备选 |
|-----|------|------|
| LLM Provider | MiniMax API (MiniMax-M2.7) | OpenAI GPT-4、Claude |
| Embedding | Ollama (bge-m3) | OpenAI Embeddings、Minimax Embed |
| 文档解析 | pdf-parse (PDF)、marked (Markdown)、mammoth (DOCX) | - |

**选型理由：**
- MiniMax M2.7：性价比高，中文理解能力强，适合IT场景
- Ollama bge-m3：本地部署 embedding，零API成本，隐私友好
- ChromaDB：开箱即用的向量库，支持RRF混合检索

### 3.6 开发工具链

| 工具 | 用途 |
|-----|------|
| concurrently | 开发时并行启动ChromaDB + Next.js |
| Python scripts | ChromaDB启动脚本、知识库同步脚本 |
| ESLint + TypeScript | 代码质量检查 |
| Prisma Migrate | 数据库迁移管理 |

### 4.2 RAG 问答流程详解

```
用户提问
    │
    ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 1: 问题理解与Embedding                                  │
│  - 用户问题 → Ollama bge-m3 → 768维向量                       │
└──────────────────────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 2: 混合检索 (ChromaDB)                                  │
│  - 向量检索: cosine similarity Top-K                          │
│  - BM25检索: 关键词匹配 Top-K                                 │
│  - RRF融合: Reciprocal Rank Fusion 合并排名                  │
└──────────────────────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 3: 上下文拼接                                            │
│  - 取Top-5相关Chunk                                           │
│  - 组装Prompt: [系统提示] + [知识上下文] + [用户问题]          │
└──────────────────────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 4: LLM生成 (MiniMax-M2.7)                               │
│  - 流式返回AI回答                                             │
│  - 同时计算置信度                                              │
└──────────────────────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 5: 置信度分级处理                                        │
│  ┌─────────────────────────────────────────────────────────┐ │
│  │ Confidence ≥ 0.8 → 直接呈现AI回答                        │ │
│  │ Confidence 0.6~0.8 → 呈现回答 + "建议转人工"提示         │ │
│  │ Confidence < 0.6 → 不显示AI回答 + 自动创建工单           │ │
│  └─────────────────────────────────────────────────────────┘ │
└──────────────────────────────────────────────────────────────┘
    │
    ▼
┌──────────────────────────────────────────────────────────────┐
│  Step 6: 工单创建 (如需要)                                     │
│  - 创建Ticket记录                                             │
│  - 保存首条Message                                            │
│  - 设置status = OPEN                                          │
└──────────────────────────────────────────────────────────────┘
```

### 4.3 置信度计算模型

```
Confidence = 0.5 × RetrievalScore + 0.3 × AnswerScore + 0.2 × CoverageScore

其中：
- RetrievalScore: ChromaDB相似度 + BM25关键词覆盖率的综合得分
- AnswerScore: 回答中知识引用完整度、避免幻觉的程度
- CoverageScore: 问题关键词在回答中的覆盖程度
```

## 5. 功能模块划分

### 5.1 模块总览

```
IT Helpdesk 系统
├── M1: 员工AI问答模块
├── M2: 工单管理模块
├── M3: 坐席工作台模块
├── M4: 知识库管理模块
├── M5: QA报告生成模块
├── M6: 反馈评价模块
└── M7: 认证授权模块
```

### 5.2 M1: 员工AI问答模块

**功能描述：** 员工通过自然语言提问，AI基于知识库即时回答。

**核心功能：**
- `F1.1` 实时聊天界面（流式响应）
- `F1.2` RAG问答引擎（混合检索 + LLM生成）
- `F1.3` 置信度评估与分级展示
- `F1.4` 低置信度自动转工单
- `F1.5` 多轮对话上下文记忆

**关键交互：**
- 用户输入问题 → API `/api/chat` → 返回AI回答 + confidence
- confidence < 0.6 时，隐式创建工单

**数据流：**
```
Question → Embedding → ChromaDB Hybrid Search → Context → MiniMax LLM → Response
```

### 5.3 M2: 工单管理模块

**功能描述：** 工单的全生命周期管理。

**工单状态流转：**
```
OPEN → AI_ANSWERED → IN_PROGRESS → RESOLVED → CLOSED
  │         │              │           │          │
  │         │              │           │          └── 评价完成，流程终结
  │         │              │           └── 坐席确认解决
  │         │              └── 坐席处理中
  │         └── AI已作答（低置信度或用户继续追问）
  └── 新建工单
```

**核心功能：**
- `F2.1` 工单创建（手动/自动）
- `F2.2` 工单列表与筛选（按状态、时间、关键词）
- `F2.3` 工单详情查看（含消息历史）
- `F2.4` 工单状态更新
- `F2.5` 工单指派（分配给坐席）

### 5.4 M3: 坐席工作台模块

**功能描述：** IT坐席的日常工作入口。

**核心功能：**
- `F3.1` 坐席仪表盘（待处理工单统计、今日工作量）
- `F3.2` 工单领取与批量分配
- `F3.3` 工单处理（回复用户、上传附件）
- `F3.4` 工单转交与协作
- `F3.5` 快捷回复模板（预留）

### 5.5 M4: 知识库管理模块

**功能描述：** 企业IT知识库的构建与维护。

**核心功能：**
- `F4.1` 文档上传（PDF/MD/DOCX）
- `F4.2` 文档自动解析与分块
- `F4.3` 向量化入库（ChromaDB）
- `F4.4` 处理进度实时展示
- `F4.5` 文档状态管理（Pending/Processing/Processed/Failed）
- `F4.6` 文档删除与重建索引

**文档分块策略：**
- Markdown: 按 `##` 二级标题切分
- PDF/DOCX: 按段落或固定长度切分（重叠128 tokens）

### 5.6 M5: QA报告生成模块

**功能描述：** 生成服务质量评估报告。

**核心功能：**
- `F5.1` 手动创建报告（坐席填写）
- `F5.2` AI辅助报告生成（基于工单上下文）
- `F5.3` 报告模板管理
- `F5.4` 报告查看与导出（预留）

**AI报告生成Prompt结构：**
```
基于以下工单信息，生成质检报告：
- 工单标题：[ticket.title]
- 工单描述：[ticket.description]
- 对话历史：[messages列表]
- 员工反馈：[feedback]

报告要求：
1. 问题定级（P0/P1/P2/P3）
2. 问题分类
3. 解决思路
4. 预防建议
5. 禁止生成故障排查步骤
```

### 5.7 M6: 反馈评价模块

**功能描述：** 员工对工单解决结果进行满意度评价。

**核心功能：**
- `F6.1` 星级评分（1-5分）
- `F6.2` 文字评价（可选）
- `F6.3` 满意度统计分析（按坐席、按时间维度）
- `F6.4` 差评预警（预留通知机制）

### 5.8 M7: 认证授权模块

**功能描述：** 管理员和坐席的安全认证。

**核心功能：**
- `F7.1` 坐席账号密码登录
- `F7.2` JWT Token签发与验证
- `F7.3` HTTP-Only Cookie会话管理
- `F7.4` 登出与会话过期处理

**权限矩阵：**

| 操作 | 员工 | 坐席 | 管理员 |
|-----|------|------|--------|
| 发起AI问答 | ✓ | ✓ | ✓ |
| 查看我的工单 | ✓ | - | - |
| 处理工单 | - | ✓ | ✓ |
| 知识库管理 | - | ✓ | ✓ |
| 生成QA报告 | - | ✓ | ✓ |
| 查看统计数据 | - | ✓ | ✓ |
| 账号管理 | - | - | ✓ |

## 9. 风险评估及应对策略

### 9.1 技术风险

| 风险 | 等级 | 描述 | 应对策略 |
|-----|------|------|---------|
| R1: LLM回答质量不稳定 | **高** | 知识库不足或检索不准确导致回答错误 | 多级置信度过滤；人工审核低置信度回答 |
| R2: Embedding性能问题 | **中** | Ollama本地embedding速度慢 | 异步处理；批量预计算 |
| R3: ChromaDB数据丢失 | **中** | 本地ChromaDB无持久化策略 | 定期备份；迁移至云向量库 |
| R4: 多轮对话上下文溢出 | **低** | 长对话token超出限制 | 限制上下文长度；摘要压缩 |

### 9.2 业务风险

| 风险 | 等级 | 描述 | 应对策略 |
|-----|------|------|---------|
| B1: 知识库覆盖不足 | **高** | 初期知识库内容少，AI无法回答常见问题 | 上线前完成80%常见问题覆盖；快速迭代补充 |
| B2: 用户不使用AI直接打电话 | **高** | 改变用户习惯困难 | 引导文案；入口明显；快速响应体验 |
| B3: 工单量远超预期 | **中** | 坐席处理不过来 | 智能分流；优先级队列；批量处理工具 |
| B4: 差评投诉 | **低** | 用户对AI回答不满意 | 差评跟踪；人工复盘 |

### 9.3 项目管理风险

| 风险 | 等级 | 描述 | 应对策略 |
|-----|------|------|---------|
| P1: 需求变更 | **中** | 业务方中途提出新需求 | 预留buffer；变更流程控制 |
| P2: 技术选型踩坑 | **中** | 某技术栈不满足需求需换 | POC验证；备选方案准备 |

## 10. 质量保障措施

### 10.1 代码质量

| 措施 | 具体做法 |
|-----|---------|
| **TypeScript严格模式** | `tsconfig.json` 开启 `strict: true`，避免 any |
| **ESLint规范** | 集成ESLint，提交前强制检查 |
| **Prettier格式化** | 统一代码风格 |
| **Code Review** | PR必须经过至少1人Review |
| **模块单一职责** | 每个文件/模块职责明确，避免God File |

### 10.2 测试策略

| 测试类型 | 覆盖率目标 | 说明 |
|---------|-----------|------|
| 单元测试 | 核心函数100% | RAG置信度计算、文档解析 |
| 集成测试 | 关键链路覆盖 | API端到端测试 |
| E2E测试 | Happy Path | Playwright/Cypress |
| 手动测试 | 全功能 | 上线前验收 |

**测试金字塔：**
```
        /\
       /  \
      / E2E \        ← 少量，核心流程
     /────────\
    /Integration \   ← API层测试
   /──────────────\
  /    Unit Tests   \ ← 大量，核心函数
 /──────────────────\
```

### 10.3 RAG质量保障

| 措施 | 说明 |
|-----|------|
| **检索评估** | 定期抽查Top-K检索结果与问题的相关性 |
| **回答审核** | 低置信度(<0.6)回答自动标记人工审核 |
| **知识库健康度** | 监控知识库Chunk覆盖率，更新滞后告警 |
| **A/B测试** | 尝试不同Prompt模板，对比回答质量 |

### 10.4 数据质量

| 措施 | 说明 |
|-----|------|
| **Prisma迁移规范** | Schema变更必须走Migration，不可直接操作DB |
| **输入校验** | API层对所有输入进行Schema校验 |
| **敏感数据保护** | 密码bcrypt哈希，JWT存HTTP-Only Cookie |
| **数据备份** | PostgreSQL定期备份，ChromaDB导出备份 |

### 10.5 监控与日志

| 维度 | 指标 |
|-----|------|
| **业务监控** | 日问答量、工单创建量、解决率、满意度 |
| **系统监控** | API响应时间、错误率、数据库连接 |
| **AI质量监控** | 平均置信度、低置信度占比、人工介入率 |

## 12. 附录

### 12.1 技术术语表

| 术语 | 全称 | 解释 |
|-----|------|------|
| RAG | Retrieval-Augmented Generation | 检索增强生成，结合知识库检索与LLM生成 |
| BM25 | Best Matching 25 | 基于关键词的经典检索算法 |
| RRF | Reciprocal Rank Fusion | 倒数排名融合，多检索结果合并策略 |
| Embedding | 文本向量化 | 将文本转为稠密向量用于相似度计算 |
| Chunk | 文档块 | 将长文档切分的最小检索单元 |
| Confidence | 置信度 | AI回答的可信程度评估 |


### 12.3 竞品对比（可选参考）

| 方案 | 优点 | 缺点 | 适用场景 |
|-----|------|------|---------|
| **自建 RAG**（本案） | 定制灵活、数据安全 | 运维成本 | 中小企业 |
| **LangChain + Pinecone** | 生态成熟 | 成本高 | 中大型企业 |
| **GPTs + Knowledge Base** | 零代码 | 定制有限、数据出境 | 快速验证 |
| **客服系统（如Intercom）** | 开箱即用 | 定制有限 | 追求快速上线 |
